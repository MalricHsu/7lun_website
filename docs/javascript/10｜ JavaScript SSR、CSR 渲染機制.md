### 一、網頁渲染（Rendering）的定義

- **渲染** 指的是把 **HTML、CSS、JavaScript** 等程式碼進行綜合解析處理，並最終生成**使用者在螢幕上實際看到的網頁畫面之過程**。
- 瀏覽器會解析程式碼、建立 DOM 樹、套用 CSS 樣式規則、執行 JavaScript，最後呈現出 UI 畫面。

#### 1. 為什麼前端需要了解網頁渲染？

- **影響使用者體驗 (UX)**：
  - 前端的核心是提供順暢的體驗。網頁渲染方式決定了：使用者要多久才能看到第一個畫面？是否會經歷長時間的白畫面？是否需要下載超大體積的 JavaScript 才能開始與網頁互動？
- **影響效能與伺服器成本**：
  - 渲染方式不同，代表運算負擔在不同的地方：
    - **CSR（瀏覽器計算）**：伺服器負擔極低，但使用者端的硬體裝置運算負擔大。
    - **SSR（伺服器計算）**：畫面顯示速度快，但伺服器運算負擔高，成本相對增加。
- **影響搜尋引擎最佳化 (SEO)**：
  - 搜尋引擎爬蟲（如 Googlebot）抓取網頁主要依賴讀取 HTML 文件。
  - 若 HTML 是空的，內容完全依賴 JavaScript 在瀏覽器端生成（純 CSR），則爬蟲可能無法抓取到完整網頁內容，會對內容網站或電商的搜尋曝光造成影響。
- **決定架構與框架選擇**：
  - 現代前端框架（如 React, Next.js, Vue, Nuxt）提供多種渲染模式（如 CSR, SSR, SSG, ISR），這會直接影響專案的目錄結構、API 呼叫時機以及部署方式。

#### 2. 靜態渲染（Static Rendering）

- **概念**：
  - HTML 內容是**事先產生好**的。
  - 網站在部署或建置時，就已經將所有頁面「編譯成實體 HTML 檔案」，存放在伺服器或 CDN。
  - 使用者發送請求時，伺服器 **直接回傳現成的 HTML**，不需再查詢資料庫或動態組裝 HTML。
- *生活比喻*：像便利商店預先做好的便當，客人點餐時直接拿給他，不需現場開火現煮。
- **前端學習對應**：將寫好的**純 HTML/CSS** 網頁放到 GitHub Pages 上託管，即為典型的靜態渲染。
- **特性**：
  - **載入速度極快**：因為 HTML 已經準備好，伺服器不需多做運算。
  - **有利於 SEO**：爬蟲在一開始就能讀取到完整的 HTML 標籤內容。
  - **成本極低、非常穩定**：幾乎不消耗任何後端計算資源。

#### 3. 動態渲染（Dynamic Rendering）

- **概念**：
  - HTML 內容是在使用者發送「**請求當下**」才即時生成的。
  - 伺服器會依據不同的請求（如登入狀態、不同使用者身分或 API 參數）來動態產生相應的 HTML 檔案。
  - 依據「**在哪裡生成 HTML 內容**」，可分為兩類：
    1. **伺服器端渲染 (Server-Side Rendering, SSR)**：在**伺服器端**即時向資料庫取得資料、組合資料與 HTML 模板，最後將完整的 HTML 字串傳送給瀏覽器。
    2. **客戶端渲染 (Client-Side Rendering, CSR)**：初始回傳的 HTML 只是空殼。由**瀏覽器執行 JavaScript**，向後端發送 API 請求取得資料，再使用 JS 動態將畫面於使用者裝置上渲染出來。
- **前端學習對應**：在 JavaScript 直播班中，使用 axios 串接 API → 取得回傳 JSON 資料 → 動態操作 DOM 渲染列表，這就是典型的 CSR。
- **特性**：
  - 能建構出**高度互動、即時資料更新**的複雜應用程式（如後台管理系統、個人社群頁面）。
  - **SSR**：首頁畫面顯示快，但伺服器負擔高。
  - **CSR**：伺服器運算負擔低，但使用者首次點入可能會有短暫白畫面，且載入 JS Bundles 時間較長。

  ![動態渲染流程](/img/js10-2.png)

#### 4. SSR（Server-Side Rendering）伺服器端渲染

- **概念**：由**伺服器端**即時取得資料並組合 HTML 模板。

  ![SSR伺服器端渲染流程](/img/js10-3.png)

- **特點**：
  - 網頁渲染主要在伺服器端進行，當資料載入完成並組合好後，才輸出完整 HTML 予瀏覽器，瀏覽器僅需負責將 HTML 標記顯示出來。
- **優點**：
  - 使用者能立即看到完整的 HTML，初始載入與畫面渲染速度快。
  - 利於 SEO，搜尋引擎爬蟲能輕易爬取完整內容。
- **缺點**：
  - 伺服器效能壓力大。若有 100 個使用者同時訪問，伺服器就需要進行 100 次運算與 HTML 渲染。
  - 前後端程式碼易緊密耦合，維護難度高。

#### 5. CSR（Client-Side Rendering）客戶端渲染

- **概念**：由 **客戶端（瀏覽器）** 執行 JavaScript 進行資料與 HTML 模板的組合。

  ![CSR客戶端渲染流程](/img/js10-4.png)

- **特點**：
  - 伺服器端回傳一個幾乎為空殼的 `index.html`，隨後瀏覽器下載並解析 JS 檔案，發送 AJAX 請求取得資料，再利用 JS 於客戶端動態組合並渲染 DOM。
- **優點**：
  - 減輕伺服器運算負擔。伺服器僅需靜態回傳 HTML/JS 檔案，並以 API 提供資料。前後端職責明確，實現**前後端分離**。
  - 互動體驗流暢，切換路由與畫面時無需整頁重新整理。
- **缺點**：
  - 首頁渲染速度較慢，必須等待 JS 下載並發送非同步網路請求取得資料後，畫面才會呈現。
  - 不利於 SEO，初始 HTML 內容空白。

### 二、SPA（Single Page Application）單頁應用程式

- 隨著渲染方式轉向由 JavaScript 主導的 **CSR (Client-Side Rendering)**，前端的角色從僅僅製作網頁切版，進化到需要負責「**頁面切換**」與「**前端路由**」，這催生了 **SPA (Single Page Application)**。

#### 1. 路由方式的差異（MPA vs SPA）

- **傳統 MPA (Multi-Page Application) 多頁應用程式**：
  - 在傳統 SSR 架構中，**每個頁面都是一個實際存在的 HTML 檔案**。
  - 使用者訪問 `/index` → 伺服器回傳 `index.html`；訪問 `/products` → 伺服器回傳 `products.html`。
  - **特點**：每次切換頁面都會重新向伺服器要一份新的 HTML。網頁由「多個頁面檔案」組成，因此稱為 MPA（多頁應用）。

- **現代 SPA (Single Page Application) 單頁應用程式**：
  - 在 CSR 架構中，**伺服器在全站僅提供一個 `index.html`**。
  - 當使用者切換 `/products`、`/cart` 等路徑時，**瀏覽器並不會重新整理網頁向伺服器要 HTML**。
  - **特點**：前端 JavaScript 會監聽 URL 的改變，依據不同的路徑狀態，利用 JS 銷毀舊的 DOM 元件並渲染新的 DOM 元件，從而實現無重整的頁面切換。
  - 提供宛如原生 App 的流暢切換體驗。

#### 2. 現代 CSR / SPA 路由方式

- 伺服器只給一個 index.html 檔案，body 內容通常只有一個 `<div id="app"></div>` 或 `<div id="root"></div>` 的空容器。
- 藉由前端 JS 進行狀態、路由與 API 的全面控制。

#### 3. SPA 誕生的核心需求

- **希望提供流暢、不重整畫面的頁面切換體驗**（如同 App 的操作感）。
- **前端能自行串接 API 取得資料並渲染**（不需依賴後端伺服器在每次點擊時重新組裝 HTML 網頁傳回）。
- **結果**：前端開發範疇大幅度擴增，需要管理 UI 狀態、路由狀態與資料處理，前端工程師的責任變得更大、技術也更多元。



### 三、使用前端框架的原因

- 當前端工程師需要以非同步方式自行處理**路由管理、資料請求、全域狀態同步與畫面效能優化**時，程式碼的模組化與管理就變得至關重要。為了提升效率並降低維護成本，前端框架（如 React, Vue）應運而生。
- **前端框架幫我們處理解決的痛點**：
  - **前端路由管理**（如 React 中的 **React Router**）。
  - **畫面狀態與資料同步**（藉由響應式狀態，資料改變時畫面自動更新，如 React 的 `useState`）。
  - **元件化開發**（模組化拆分，例如將導覽列、卡片拆成獨立 Function Component 方便重複使用）。
  - **生命週期管理**（在元件載入、更新、銷毀的不同階段執行對應操作，如 `useEffect`）。
  - **效能優化**（使用 **Virtual DOM** 自動比對新舊畫面的差異，只針對有變動的實體 DOM 進行局部更新，提升網頁繪製效能）。
